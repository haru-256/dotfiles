# AGENTS.md

このファイルはグローバルに利用する共通ガイドです。リポジトリ固有の `AGENTS.md`、システム指示、ユーザー指示がある場合は、より具体的で制約の強いものを優先してください。

---

## 優先順位

より近い階層のルールが優先されます。矛盾が生じた場合、セキュリティ・安全側のルールを優先してください。

```text
リポジトリ固有 AGENTS.md
  > システム指示 / ユーザー指示
    > SECURITY.md / CI 設定 / CODEOWNERS
      > このファイル
```

---

## 作業前の確認

- `git status` と必要に応じて `git diff` で未コミット変更を把握する
- 変更対象の関連ファイルと周辺実装を確認し、既存の設計・命名・構成を把握する
- 同じ責務の既存ヘルパーやパターンがあれば、新規実装より再利用を優先する
- ツール・ライブラリ・API など外部情報が必要な場合は、Web 検索で最新情報を確認してから判断する

---

## 変更方針

**必要十分に留める**
- 依頼された目的を最短で満たす変更のみを行う
- 無関係なリファクタリング・整形・スタイル修正はしない
- 既存の設計・命名・構成・フォーマットに合わせる
- 変更理由がある場合を除き、既存挙動との互換性を保つ

**実装の質**
- エラーを握りつぶさず、既存の流儀に沿って明示的に扱う
- 一時しのぎより根本原因の解消を優先する
- コメントは意図が読み取りにくい箇所にだけ、簡潔に付ける

**変更範囲**
- 依頼範囲に限定しつつ、成立に必要な関連箇所は漏れなく更新する
- 新しいファイルや設定を追加する場合は、既存構成との整合性を保つ
- 既存ファイルの移動・削除は、必要性が明確な場合だけ行う
- ガバナンス文書・CI 設定・権限設定の変更は、明示的に依頼された場合だけ行う

## 不明点・判断に迷う場面

**止まる前に自問する**

1. 既存コードに同様のパターンはあるか？ → あれば従う
2. 依頼の意図から合理的に推測できるか？ → できれば進める
3. 取り返しのつかない変更か？ → そうなら確認を取る

**確認が必要な場合**

- コンフリクトや想定外の差分を発見したとき
- 依頼範囲外への影響が広範囲になるとき
- 破壊的な操作（削除・移動・スキーマ変更など）が必要なとき

---

## コマンド実行

- コマンドが見つからない場合は、まず `mise` で管理されていないかを確認する
- `mise` 管理下なら `mise exec` 経由で実行する
- 未インストールなら `mise` での導入を提案する
- ファイル検索・パターン検索には `rg`・`fd` などの高速ツールを積極的に使う

## Git 運用

**日常操作**
- ユーザーが作成した未コミット変更は勝手に巻き戻さない
- 明示的に依頼されていない限り、commit・push・tag・release・merge は行わない
- `main`・`master`・保護ブランチへの直接変更・直接 push は行わない
- 履歴改変を伴う操作（`reset --hard`・`rebase`・`push --force`）は、明示的な依頼なしでは行わない

**コミットメッセージ**（Conventional Commits）

```
<type>(<scope>): <subject>
```

| type | 用途 |
|------|------|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `refactor` | 機能追加・バグ修正を伴わない構造改善 |
| `test` | テストコードの追加・修正 |
| `chore` | ビルド設定・依存関係の更新 |

scope は変更の影響範囲を括弧内に示す（例：`feat(parser): arXiv フィードの解析ロジックを追加`）

**PR 作成時**
- リポジトリ内の PR テンプレートを先に確認し、テンプレートがあればそれに従う

---

## 安全性

- 機密情報・個人情報・認証情報を追加・出力・コミットしない
- シークレット・環境変数ファイル・アクセス制御の変更は最小限に留める
- 破壊的な操作は、明示的に求められた場合を除いて避ける
- 変更理由・意図を差分や説明で明確にし、作業内容が追跡可能な状態を保つ

---

## 検証

- 変更に関係する既存のテスト・ビルド・lint を活用する
- 設定変更・スクリプト変更では、実際の利用経路に沿って影響を確認する
- 検証に失敗した場合は、成功したように扱わず、失敗内容と制約を明示する

---

## ドキュメント

- 使い方や前提が変わる変更では、必要な範囲で関連ドキュメントを更新する
- `AGENTS.md` 自体の変更は、通常のコード変更より慎重に扱い、意図が明確な場合に限る
