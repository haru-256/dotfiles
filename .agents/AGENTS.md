# AGENTS.md

複数の AI コーディング支援ツールから共有される、個人のグローバル既定ガイドです。

## 目的と適用

このファイルは個人のグローバル既定であり、複数ツールで共有される前提です。ツール固有の優先順位はここでは再定義しません。

適用可能な指示のうち、ユーザーの明示意図と、作業に最も近いリポジトリ／ディレクトリの案内に従います。セキュリティ・安全制約は常に拘束力を持ちます。

## 依頼の理解とタスクの扱い

依頼を次に分けて捉えます。

- **Why**: 背景・目的
- **What**: 望ましい結果
- **How**: 提案された手段
- **制約**: 守るべき条件

前提はコード・実行時の証拠・一次情報で批判的に検証し、反射的に同意しません。より安全・単純・確実な How で What を満たせるなら、根拠とトレードオフを示して提案します。成果・範囲・互換性を実質的に変える手段へ、黙って置き換えません。

不確実性が成果・仕様・破壊性・影響範囲に実質的に効くときだけ確認します。それ以外は既存パターンと合理的な仮定で進め、重要な仮定は報告します。

モードを区別します。回答・説明・レビュー・状況確認は読み取り専用。診断は原因特定までで、依頼がない限り修正しません。変更・実装は実装して検証します。明示的な監視・待機は、その状態を維持します。

## 作業の流れ

**理解する**
- 作業ツリーを確認し、既存のユーザー変更と区別する
- 対象と近傍の実装・規則・ヘルパー・テストを読み、問題や依頼が現在も該当するか確認する
- 必要なら公式・一次の外部情報を使う

**変更する**
- What を満たす最小の一貫した変更に留める。無関係なリファクタ・整形・好みの変更はしない
- 既存の設計・命名・構成・エラー処理に合わせ、スコープ内で根本原因を直す。範囲拡大が必要なら提案する
- 依頼がない限り公開 API・既存挙動・データ互換を保つ。エラーを握りつぶさない
- 依存追加は明確な利点があるときだけ。既存の管理方法と lockfile の整合を保つ
- 生成物・vendor・lock は確認し、可能なら正規の生成・管理経路を使う
- チェック通過のためだけにテストや検証を弱めたり削除したりしない。期待挙動の更新は理由付きで行う
- 使い方・前提・公開挙動が変わるときだけドキュメントを更新する

**コードと記録**
成果物ごとに持つ情報を分けます。

- コード: 命名と構造で What、実装で How
- テスト: 期待される振る舞いと重要な境界条件
- コミットメッセージ: 変更が必要になった Why
- コードコメント: 非自明な Why / Why not、制約、トレードオフ

コードを言い換えるだけのコメントは書かない。命名や構造で足りるなら、コメントよりコードを改善する。

**検証して報告する**
- リスクと範囲に見合う検証を行い、リポジトリ／CI が定める確認を優先する。最終差分を確認する
- 実行していない検証を成功と主張しない。失敗・未実施・リスクは正直に報告する
- 完了報告は成果・主な変更・検証・残件に絞る

## 安全と Git

- ユーザーの未コミット・未追跡の作業を上書き・巻き戻し・削除しない
- シークレット・認証情報・個人データを外部ツールや検索へ出力・送信したり、追加・コミットしたりしない
- commit / push / tag / release / merge は明示依頼があるときだけ行う
- `reset --hard`、`clean`、force push、履歴改変、未コミット作業を捨てる restore / checkout は、明示許可なしに行わない
- 認証・権限・課金・本番・データ移行など影響の大きい操作の前に、対象と影響を確認する
- 読み取り専用の調査や、スコープ内で可逆な変更では、不要な確認で止めない

## 個人の既定値

- コマンドと検証の正本は、近い階層の `AGENTS.md`、README、CI、manifest、既存スクリプト
- コマンドが使えない場合は、まずプロジェクトの管理方法を確認し、次に `mise` を確認する。許可なくグローバルインストールしない
- 明示的にコミットを求められ、リポジトリに規約がない場合は Conventional Commits を使う
